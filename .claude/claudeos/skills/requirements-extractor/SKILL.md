# requirements-extractor

## 概要

議事録、要件メモ、メール、Slack ログなどの**非構造化テキスト**から
実行可能タスクと受け入れ基準を抽出し、Issue 化できる形に整形する skill。

## 使う場面

- 議事録から TODO を拾って Issue 化したい
- 要件説明メール / PRD ドキュメントを実装可能なタスクに分解したい
- 1 on 1 / レビュー記録から改善 Issue を起こしたい
- 要件と現実装のギャップを GAP 一覧として出力したい

## 想定入力

- ユーザが貼り付ける議事録テキスト、要件文書、メール本文、Slack エクスポート
- Read で指定されたローカル文書（`docs/requirements.md` 等）
- ユーザが「この議事録からタスクを起こして」と依頼

## 期待する出力

各タスクに対して以下を構造化して提示する:

```yaml
- title: "feat: 〇〇機能の追加"           # GitHub Issue 用タイトル
  type: feat | fix | docs | refactor | chore
  priority: P1 | P2 | P3
  acceptance:                              # 受け入れ基準
    - 基準 1
    - 基準 2
  rationale: |
    なぜこの作業が必要か（議事録から引用）
  source_quote: "（議事録の該当発言を 1-2 文で引用）"
  estimate: 30m | 2h | 1d                  # 任意
  blocked_by: []                           # 依存タスクがあれば
  related_files: []                        # 想定変更箇所
```

抽出後、ユーザに「これらをそのまま `gh issue create` で起票しますか？」と確認する。

## 標準手順

1. 入力テキストの全文を一度読み、議論の論点と決定事項を把握する
2. **決定事項のみ** をタスクとして抽出する（「議論があった」だけのものは除外）
3. 各タスクを type / priority で分類する
4. 受け入れ基準を **検証可能な文言** で書く（「ちゃんと動く」は NG、「POST /foo が 201 を返す」は OK）
5. P1 優先で並べ、blocked_by 関係を整理する
6. ユーザに提示し、Issue 起票するか確認する

## 詳細ガイド

### 抽出するもの

- 明示的な TODO / 決定事項
- 受け入れ基準として書ける具体的要望
- 期日や担当者が明示されたもの
- 質問への回答で「こうする」と決まったもの

### 抽出しないもの

- 単なる議論・意見交換
- 「検討する」「考えておく」レベルの未確定事項
- 既に完了している報告
- 雑談・社内事情

### Priority 判定基準

| 優先度 | 判定材料 |
|---|---|
| P1 | 「至急」「緊急」「ブロッカー」「今週中」明記 / セキュリティ / 顧客影響 |
| P2 | 「次スプリント」「来月まで」 / UX 改善 / テスト追加 |
| P3 | 「いつか」「余裕があれば」 / 軽微なリファクタ |

### Issue 化のコマンド例

```bash
gh issue create \
  --title "feat: 〇〇機能の追加" \
  --label "P1,feat" \
  --body "$(cat <<'EOF'
## 解決したい課題
…
## 受け入れ基準
- [ ] …
## 参考
> （議事録引用）
EOF
)"
```

## 連携

- agent: `product-manager` / `architect`（タスク分解の妥当性レビュー）
- command: `/extract-tasks <file>` から起動する想定
- Issue Factory: §7 の Issue 生成条件の **入力テキスト経路**として位置づける
